## 2-6. 연습문제

~~~
✅ 학습 목표 :
* 연습문제(7문제 이상) 푼 것들 정리하기
~~~

이번 주차에서는 문제를 직접 손으로 풀어보는 과정이 특히 중요하다고 느꼈습니다. 단순히 정답 쿼리를 보는 것보다, 먼저 제가 스스로 쿼리를 작성해보고 강의 풀이와 비교해보니 제가 어떤 개념에서 막히는지 더 분명하게 확인할 수 있었습니다.

제가 직접 풀어본 문제는 **1번, 2번, 5번, 7번, 10번, 11번, 13번, 14번, 17번**입니다. 이 중 특히 배운 점이 많았던 문제들을 중심으로 정리하였습니다.

### 1) 1번 문제: 타입2가 없는 포켓몬의 수 구하기
제가 처음에는 `type2 = null`처럼 비교하려고 생각하였습니다. 그런데 강의 풀이와 비교해보니 null은 일반 값처럼 `=`로 비교하는 것이 아니라 `IS NULL`로 확인해야 한다고 이해했습니다. 

또한 null은 0이나 빈 문자열과도 다르며, **아예 값이 없는 상태**라고 이해했습니다. 이 문제를 통해 SQL에서 null 처리가 왜 별도로 중요한지 알게 되었습니다.

```sql
SELECT COUNT(id) AS cnt
FROM pokemon
WHERE type2 IS NULL;
```

이 문제에서 저는 **조건을 먼저 해석하고, 그 다음에 집계 방식을 정하는 순서**가 중요하다고 느꼈습니다.

### 2) 2번 문제: 타입2가 없는 포켓몬의 type1별 수 구하기
이 문제에서는 저는 `COUNT(id)`만 쓰고 바로 결과를 얻으려고 했는데, `type1`을 함께 조회하려면 `GROUP BY type1`이 반드시 필요하다는 점을 다시 확인하였습니다. 

즉, **집계 기준 컬럼이 존재하면 GROUP BY를 반드시 써야 한다**고 이해했습니다. 또한 문제에서 “큰 순으로 정렬”이라고 했기 때문에 `ORDER BY ... DESC`까지 함께 써야 완성된 쿼리라는 점도 배웠습니다.

```sql
SELECT type1, COUNT(id) AS pokemon_cnt
FROM pokemon
WHERE type2 IS NULL
GROUP BY type1
ORDER BY pokemon_cnt DESC;
```

이 문제를 통해 저는 SQL에서 **조건, 집계, 정렬이 각각 다른 절에서 처리된다**는 구조를 더 분명히 이해하게 되었습니다.

### 3) 5번 문제: 동명이인이 있는 이름 찾기
이 문제는 제가 가장 헷갈렸던 문제 중 하나였습니다. 처음에는 `WHERE COUNT(name) >= 2`처럼 작성하려고 생각하였는데, 강의 풀이를 보면서 이 방식은 틀렸다고 이해했습니다. 

`WHERE`는 원본 데이터에 조건을 거는 절이고, `COUNT(name)`처럼 집계한 결과에 조건을 거는 것은 `HAVING`을 사용해야 하였습니다. 이 차이를 이번 문제를 통해 확실히 구분할 수 있었습니다.

```sql
SELECT name, COUNT(name) AS trainer_cnt
FROM trainer
GROUP BY name
HAVING trainer_cnt >= 2;
```

이 문제를 통해 **WHERE는 집계 전, HAVING은 집계 후**라는 원칙을 실제 문제 안에서 이해하였습니다. 또한 서브쿼리로도 풀 수 있지만, 이 경우에는 HAVING을 쓰는 것이 더 간결하다고 느꼈습니다.

### 4) 7번 문제: 특정 3명의 트레이너 정보 조회하기
이 문제를 처음 풀 때는 `OR`를 여러 번 연결해서 작성하였습니다. 물론 결과는 얻을 수 있었지만, 강의 풀이를 보니 `IN`을 쓰는 것이 훨씬 짧고 읽기 쉽다고 느꼈습니다.

```sql
SELECT *
FROM trainer
WHERE name IN ('아이리스', '다운', '크리스');
```

이 문제를 통해 **같은 컬럼에서 여러 값을 조건으로 걸 때는 IN이 가독성과 효율 면에서 더 좋다**고 배웠습니다. 또한 `OR`를 많이 쓸 경우에는 괄호를 잘 써야 연산 우선순위가 꼬이지 않는다는 점도 함께 배웠습니다.

### 5) 10번 문제: 타입2가 존재하는 포켓몬 수 구하기
1번 문제와 반대로 이 문제에서는 null이 아닌 값을 확인해야 했습니다. 그래서 `IS NOT NULL`을 사용해야 한다고 이해했습니다.

```sql
SELECT COUNT(id) AS pokemon_cnt
FROM pokemon
WHERE type2 IS NOT NULL;
```

이 문제는 1번 문제와 짝을 이루는 문제처럼 느껴졌습니다. `IS NULL`과 `IS NOT NULL`을 비교하면서, null 조건은 SQL에서 매우 자주 나오는 기본 패턴이라고 생각했습니다.

### 6) 11번 문제: 타입2가 있는 포켓몬 중 가장 많은 type1 찾기
이 문제에서는 단순히 집계만 하는 것이 아니라, **가장 많은 값 하나만 뽑아야 한다**는 점이 중요했습니다. 저는 처음에 `ORDER BY`까지만 떠올렸는데, 강의에서는 `LIMIT 1`까지 써서 원하는 결과 하나만 남겼습니다.

```sql
SELECT type1, COUNT(id) AS pokemon_cnt
FROM pokemon
WHERE type2 IS NOT NULL
GROUP BY type1
ORDER BY pokemon_cnt DESC
LIMIT 1;
```

이 문제를 통해 `ORDER BY`는 정렬만 해줄 뿐이고, **정말 하나만 보고 싶을 때는 LIMIT까지 써야 한다**고 이해했습니다.

### 7) 13번 문제: 이름에 ‘파’가 들어가는 포켓몬 찾기
이 문제는 문자열 검색 문법을 익히는 데 도움이 되었습니다. 저는 처음에는 정확히 일치하는 경우만 생각했는데, 강의 풀이를 보면서 `LIKE`와 `%`의 조합이 중요하다고 배웠습니다.

```sql
SELECT korean_name
FROM pokemon
WHERE korean_name LIKE '%파%';
```

이 문제를 통해 다음과 같이 정리하였습니다.

- `파%` : 파로 시작하는 값
- `%파` : 파로 끝나는 값
- `%파%` : 파가 포함된 값

즉, 문자열 검색에서도 조건을 얼마나 구체적으로 쓰느냐가 중요하다고 느꼈습니다.

### 8) 14번 문제: 배지가 6개 이상인 트레이너 수 구하기
이 문제는 비교적 무난하게 풀었지만, 숫자 조건을 걸고 카운트를 하는 가장 기본적인 구조를 복습할 수 있었습니다.

```sql
SELECT COUNT(id) AS trainer_cnt
FROM trainer
WHERE badge_count >= 6;
```

이 문제를 풀면서 SQL 문제를 볼 때 복잡하게 생각하기보다, 먼저 **테이블, 조건, 집계 여부**를 나눠서 보는 습관이 중요하다고 다시 느꼈습니다.

### 9) 17번 문제: 풀어준 포켓몬 비율이 20%가 넘는 트레이너 찾기
이 문제는 이번 주차에서 가장 어려웠던 문제였습니다. 단순 count가 아니라 **조건을 만족하는 개수 / 전체 개수**라는 비율 계산이 들어갔기 때문입니다. 

강의를 통해 `COUNTIF()`를 처음 배웠고, 조건을 만족하는 행만 세는 방식이 매우 유용하다고 느꼈습니다.

```sql
SELECT
  trainer_id,
  COUNTIF(status = 'Released') AS released_cnt,
  COUNT(pokemon_id) AS pokemon_cnt,
  COUNTIF(status = 'Released') / COUNT(pokemon_id) AS released_ratio
FROM trainer_pokemon
GROUP BY trainer_id
HAVING released_ratio > 0.2;
```

이 문제를 통해 저는 다음을 배웠습니다.

1. `COUNTIF()`는 조건을 만족하는 행만 바로 세어줄 수 있습니다.  
2. 집계 결과로 만든 비율에도 `HAVING`으로 조건을 걸 수 있습니다.  
3. 문제를 한 번에 풀기 어렵다면, 먼저 분자와 분모를 각각 구한 뒤 마지막에 나누는 방식으로 단계적으로 접근하는 것이 좋다고 느꼈습니다.

5. 마지막으로 정렬과 제한 조건이 있는지 확인하였습니다.

## 2-8. 새로운 집계함수

~~~
✅ 학습 목표 :
* SQL 쿼리 구조를 이해할 수 있다. 
* SELECT, FROM, WHERE을 활용하는 방법을 설명할 수 있다. 
~~~

### 1) SQL 쿼리의 기본 구조
저는 SQL의 기본 구조를 아래와 같이 정리하였습니다.

- `SELECT` : 보고 싶은 컬럼이나 집계값을 선택하는 부분입니다.  
- `FROM` : 어떤 테이블에서 데이터를 가져올지 정하는 부분입니다.  
- `WHERE` : 원본 데이터에서 먼저 조건을 거는 부분입니다.  
- `GROUP BY` : 같은 기준끼리 묶어서 요약할 때 사용하는 부분입니다.  
- `HAVING` : 집계한 결과에 다시 조건을 거는 부분입니다.  
- `ORDER BY` : 결과를 정렬하는 부분입니다.  
- `LIMIT` : 결과 행 수를 제한하는 부분입니다.  

특히 이번 주차에서는 `WHERE`와 `HAVING`의 차이를 더 분명히 이해하였습니다. `WHERE`는 집계 이전의 데이터에 적용하고, `HAVING`은 집계 이후 결과에 적용한다고 정리하였습니다.

### 2) COUNTIF에 대해 배운 점
이번 주차에서 새롭게 인상 깊었던 것은 `COUNTIF()`였습니다. 

기존에는 `COUNT()`만 익숙했는데, `COUNTIF()`는 조건을 만족하는 행만 바로 셀 수 있어서 매우 편리하다고 느꼈습니다.

예를 들어, 풀어준 포켓몬의 수를 셀 때는 아래처럼 사용할 수 있었습니다.

```sql
COUNTIF(status = 'Released')
```

이 함수는 조건이 붙은 count를 간단히 작성할 수 있게 해주기 때문에, 나중에 비율 계산이나 상태별 집계에서도 자주 사용할 것 같다고 생각했습니다.

### 3) 새롭게 알게 된 GROUP BY ALL

추가로 강의 후반부에서는 `GROUP BY ALL`이라는 기능도 새롭게 알게 되었습니다. 

기존에는 `SELECT`에 쓴 컬럼을 `GROUP BY`에 하나씩 직접 적어주어야 했는데, `GROUP BY ALL`을 사용하면 이를 자동으로 추론해준다고 이해했습니다.

예를 들어 다음과 같이 사용할 수 있습니다.

```sql
SELECT first_name, last_name, SUM(score) AS total_score
FROM player_stats
GROUP BY ALL;
```

이 기능은 반복 입력을 줄여서 편리하다고 느꼈습니다. 

다만 학습 초기에는 어떤 컬럼을 기준으로 묶는지 스스로 명확히 이해하는 것이 중요하므로, 먼저 기본적인 `GROUP BY 컬럼명` 방식에 익숙해지는 것이 필요하다고 생각했습니다.

### 4) 이번 주차에서 함께 정리한 핵심 개념

이번 강의를 통해 아래 개념들을 함께 복습하였습니다.

- `NULL`은 값이 없는 상태이며 `IS NULL`, `IS NOT NULL`로 확인합니다.  
- `DISTINCT`는 중복을 제거하고 고유값만 보고 싶을 때 사용합니다.  
- `IN`은 같은 컬럼에 여러 값을 조건으로 줄 때 유용합니다.  
- `LIKE`는 문자열 패턴 검색에 사용하며 `%`와 함께 자주 사용합니다.  
- `ORDER BY ... DESC`는 큰 값부터 내림차순 정렬할 때 사용합니다.  
- `LIMIT`은 상위 몇 개 결과만 보고 싶을 때 사용합니다.  

결과적으로 저는 SQL 쿼리 구조를 볼 때, 각 절의 역할을 분리해서 이해해야 쿼리가 훨씬 덜 복잡하게 느껴진다고 생각했습니다.


## 3-2. 쿼리를 작성하는 흐름

~~~
✅ 학습 목표 :
* 쿼리를 작성하는 흐름을 설명할 수 있다.
~~~

이번 파트에서 가장 인상 깊었던 점은, SQL을 잘 작성하는 것은 단순히 문법을 많이 아는 것만으로 해결되지 않는다는 부분이었습니다. 

강의에서는 쿼리 작성 전과 후의 과정까지 포함해서 하나의 흐름으로 설명하였고, 저는 이 부분이 실무적인 관점에서 특히 중요하다고 느꼈습니다.

제가 이해한 SQL 쿼리 작성 흐름은 다음과 같습니다.

### 1) 먼저 문제와 지표를 정의하였습니다.

무엇을 알고 싶은지 먼저 분명히 정해야 합니다. 단순히 “유저 수를 보고 싶다”가 아니라, 어떤 유저를 어떤 기준으로 셀 것인지 구체적으로 정해야 한다고 이해했습니다. 

강의에서는 지표를 구체화할 때 **분자와 분모를 함께 적어보는 방식**을 추천하였는데, 저도 이 방법이 매우 유용하다고 느꼈습니다. 정의가 애매하면 쿼리를 맞게 작성해도 결과 해석이 흔들릴 수 있기 때문입니다.

### 2) 유사한 문제와 기존 지표를 먼저 탐색하였습니다.
강의에서는 회사나 문서에 비슷한 문제가 이미 해결되어 있는지 먼저 찾아보는 것이 효율적이라고 설명하였습니다. 저도 이 부분을 듣고, 무조건 처음부터 새로 짜기보다 기존 쿼리나 기존 지표 정의를 참고하는 습관이 중요하다고 생각했습니다.

### 3) 사용할 테이블과 컬럼을 확인하였습니다.
문제를 정의한 뒤에는 어떤 테이블을 써야 하는지, 어떤 컬럼이 필요한지 확인해야 합니다. 테이블이 하나면 비교적 간단하지만, 여러 개라면 이후에 `JOIN`까지 고려해야 한다고 이해했습니다. 

또한 강의에서 반복적으로 강조한 것처럼, 데이터를 직접 미리 보면서 컬럼의 의미와 값의 형태를 파악하는 과정이 중요하다고 느꼈습니다.

### 4) 쿼리는 단계적으로 작성하였습니다.
강의에서는 처음부터 완성형 쿼리를 한 번에 쓰기보다, 아래처럼 단계적으로 접근하는 흐름을 강조하였습니다.

1. 사용할 테이블을 정하였습니다.  
2. 조건이 있는지 확인하였습니다.  
3. 조회할 컬럼을 정하였습니다.  
4. 집계가 필요한지 판단하였습니다.  
5. 정렬이나 제한 조건을 마지막에 추가하였습니다.  

저는 이 흐름이 특히 초보자에게 유용하다고 느꼈습니다. 실제로 이번 문제풀이에서도 이 순서로 생각하니 훨씬 덜 막혔습니다.

### 5) 결과의 정합성을 확인하였습니다.
강의에서 매우 중요하다고 느낀 부분은 **정답처럼 보이는 쿼리를 작성했다고 끝이 아니라는 점**이었습니다. 내가 기대한 값과 결과가 맞는지, 숫자가 이상하지 않은지, 중복이 없는지 등을 반드시 검증해야 한다고 이해했습니다.

예를 들어, count 결과가 예상 행 수와 맞는지 직접 확인하거나, distinct 여부를 비교해보는 습관이 정합성 검증에 도움이 된다고 느꼈습니다.

### 6) 가독성을 높이고 저장하였습니다.
쿼리는 한 번 쓰고 끝나는 것이 아니라 다시 보거나 다른 사람과 공유할 수 있기 때문에, 읽기 쉽게 작성하고 문서로 남겨야 한다고 이해했습니다. 

즉, 좋은 쿼리는 단순히 “실행되는 쿼리”가 아니라, **정확하고, 검증되었고, 다시 읽기 쉬운 쿼리**라고 생각했습니다.

### 정리
이번 파트를 통해 저는 SQL 쿼리 작성의 흐름을 다음 한 문장으로 정리하였습니다.

**문제를 정의하고 → 지표를 구체화하고 → 테이블을 확인한 뒤 → 단계적으로 쿼리를 작성하고 → 결과를 검증하고 → 읽기 좋게 정리하여 저장하는 과정**이라고 이해했습니다.

이전에는 SQL 작성이 문법 연습에만 가깝다고 생각했는데, 이번에는 문제 해결 과정 전체를 설계하는 일이라고 느꼈습니다.


## 3-3. 쿼리 작성 템플릿과 생산성 도구

~~~
✅ 학습 목표 :
* 생산성 도구를 만들 수 있다.
~~~

이번 파트에서는 쿼리를 잘 쓰기 위해 템플릿을 먼저 만들어두는 습관이 중요하다고 배웠습니다. 
강의에서는 매번 같은 구조를 반복해서 쓰는 대신, 생산성 도구를 이용해 템플릿을 빠르게 불러오는 방식을 소개하였습니다.

### 1) 제가 정리한 쿼리 작성 템플릿
저는 강의 내용을 바탕으로 아래 템플릿을 정리하였습니다.

```text
[쿼리 작성 전 정리]
- 커리를 작성하는 목표:
- 확인할 지표:
- 계산 방법:
- 데이터 기간:
- 사용할 테이블:
- JOIN KEY:
- 데이터 특징:

[SQL 초안]
SELECT
FROM
WHERE
GROUP BY
HAVING
ORDER BY
LIMIT
```

이 템플릿이 있으면 문제를 보자마자 바로 SQL부터 쓰는 것이 아니라, 먼저 구조를 정리한 뒤 작성할 수 있어서 훨씬 덜 헤매게 된다고 느꼈습니다.

### 2) 생산성 도구로 Espanso를 사용해보는 방식
강의에서는 `Espanso`라는 생산성 도구를 소개하였습니다. 특정 단어를 입력하면 미리 저장한 템플릿이 자동으로 펼쳐지는 방식이라고 이해했습니다.

예를 들어 `:sql` 같은 트리거를 입력하면 아래 템플릿이 자동으로 나오도록 설정할 수 있습니다.

```yaml
matches:
  - trigger: ":sql"
    replace: |
      [쿼리 작성 전 정리]
      - 커리를 작성하는 목표:
      - 확인할 지표:
      - 계산 방법:
      - 데이터 기간:
      - 사용할 테이블:
      - JOIN KEY:
      - 데이터 특징:

      [SQL 초안]
      SELECT
      FROM
      WHERE
      GROUP BY
      HAVING
      ORDER BY
      LIMIT
```

이런 방식은 반복 입력을 줄여주기 때문에, 초반에 쿼리 구조를 잡는 데 많은 도움이 될 것이라고 느꼈습니다.

### 3) 생산성 도구를 왜 써야 하는지에 대한 제 생각

처음에는 템플릿을 그냥 메모장에 적어두면 되지 않을까 생각했습니다. 
그런데 강의 내용을 듣고 보니, 템플릿은 만들어두기만 하고 실제로는 잘 안 쓰게 될 가능성이 높다고 느꼈습니다. 
그래서 **불러오는 과정까지 자동화해야 습관이 된다**고 이해했습니다.

저는 이 부분이 매우 실용적이라고 느꼈습니다. SQL 실력이 부족한 초보자일수록 템플릿을 자주 써서 문제를 구조적으로 바라보는 연습이 필요하다고 생각했습니다.

### 4) 실습 정리

이번 주차에서는 실제로 생산성 도구의 개념을 이해하고, 어떤 형식으로 템플릿을 등록하면 되는지 정리하였습니다. 

- 트리거 단어를 정합니다.  
- 템플릿 내용을 등록합니다.  
- 입력 시 자동으로 템플릿이 펼쳐지는지 확인합니다.  

이 과정을 통해 저는 단순히 SQL 문법만 배우는 것이 아니라, **SQL을 더 안정적이고 반복 가능하게 작성하는 습관**도 함께 배우고 있다고 느꼈습니다.

# 3️⃣ 확인문제

## 문제 1

> **🧚Q. Q. 포켓몬 연구에 흥미를 느낀 혜인은 각 타입(type1)별 평균 공격력(attack)을 비교해보고 싶었습니다.**
>
> 그래서 다음과 같은 필요한 정보를 미리 정리해보았습니다. 

~~~
조건 : attack이 50 이상인 포켓몬만 포함
보고 싶은 컬럼 : type1
집계 내용 : 각 type1 별 평균 공격력
정렬 기준 : 평균 공격력을 기준으로 내림차순 정렬
~~~

> **이 목표를 바탕으로 혜인은 아래와 같은 쿼리를 작성했지만, 일부 SQL 문법 요소를 빼먹었습니다. 비어 있는 부분인 ㄱ, ㄴ, ㄷ, ㄹ 에 들어갈 알맞은 SQL 구문을 채워보세요:**

~~~sql
SELECT type1, (ㄱ)
FROM pokemon
(ㄴ) attack >= 50
(ㄷ) type1
ORDER BY (ㄱ) (ㄹ);
~~~



~~~
ㄱ : AVG(attack)
ㄴ : WHERE
ㄷ : GROUP BY
ㄹ : DESC

완성된 쿼리 :
SELECT type1, AVG(attack)
FROM pokemon
WHERE attack >= 50
GROUP BY type1
ORDER BY AVG(attack) DESC;
~~~
