# 1️⃣ 개념정리

## 3-4. 오류를 디버깅하는 방법

~~~
✅ 학습 목표 :
* 오류의 정의에 대해 설명할 수 있다. 
* 오류 메시지를 보고 디버깅이라는 과정을 수행할 수 있다. 
~~~

오류는 단순히 “실패”를 뜻하는 것이 아니라, 지금 작성한 방식으로는 원하는 결과를 얻을 수 없다는 사실을 알려주는 신호라고 이해했습니다. 

강의에서는 오류 메시지를 길을 잃지 않게 도와주는 길잡이처럼 받아들이는 태도가 중요하다고 설명하였고, 저도 이 관점이 디버깅을 훨씬 차분하게 만들어준다고 생각했습니다. 

특히 오류 메시지는 대부분 문제가 발생한 위치를 함께 알려준다는 점이 중요했습니다. 

몇 번째 줄, 몇 번째 칸에서 문제가 감지되었는지를 알려주기 때문에, 막연하게 전체 쿼리를 다시 보는 것이 아니라 문제 구간을 좁혀서 확인할 수 있습니다. 

따라서 오류 메시지를 무시하지 않고, 먼저 번역하고 해석하는 습관이 필요하다고 생각했습니다.

제가 정리한 디버깅 과정은 다음과 같습니다.

| 단계 | 확인 내용 | 정리 |
| --- | --- | --- |
| 1 | 오류 메시지 읽기 | 어떤 종류의 오류인지 먼저 파악합니다. |
| 2 | 위치 확인 | 몇 번째 줄, 몇 번째 칸인지 확인합니다. |
| 3 | 문법 검토 | SELECT, FROM, WHERE, GROUP BY, 괄호, 세미콜론 순서를 점검합니다. |
| 4 | 함수 사용법 점검 | 함수 인자 개수, 자료형, 집계 여부를 확인합니다. |
| 5 | 검색 및 재확인 | 공식 문서, 검색, GPT 등의 생서형 AI를 활용해 검증합니다. |

강의에서 소개된 대표적인 오류 패턴도 함께 정리해보았습니다.

| 오류 유형 | 원인 | 해결 방법 |
| --- | --- | --- |
| `SELECT list must not be empty` | SELECT 절에 컬럼이 비어 있음 | SELECT 뒤에 컬럼을 명시합니다. |
| `Number of arguments does not match ...` | 함수에 넣은 인자 수가 잘못됨 | 함수 문법을 다시 확인합니다. |
| 집계 함수 관련 오류 | GROUP BY 없이 일반 컬럼과 집계 컬럼을 함께 사용함 | 필요한 컬럼을 GROUP BY에 추가합니다. |
| `Expected end of input but got keyword SELECT` | 여러 쿼리를 구분하지 않음 | 쿼리 끝에 세미콜론(`;`)을 붙입니다. |
| LIMIT/WHERE 순서 오류 | 절의 순서가 잘못됨 | WHERE 뒤에 LIMIT이 오도록 수정합니다. |
| 괄호 관련 오류 | 괄호를 열고 닫지 않음 | 괄호 짝을 다시 맞춥니다. |

결국 디버깅은 감으로 고치는 작업이 아니라, 오류 메시지를 해석하고 문법과 구조를 하나씩 검토하는 과정이라고 이해했습니다. 앞으로는 오류가 나면 당황하기보다, 오류 메시지를 먼저 읽고 좌표와 원인을 찾는 방식으로 접근해야겠다고 생각했습니다.



## 4-2. 데이터 타입과 데이터 변환(CAST, SAFE_CAST)

~~~
✅ 학습 목표 :
* 데이터 타입의 종류를 설명할 수 있다. 
* 데이터 타입을 변환하는 방법을 설명할 수 있다. 
~~~

데이터를 다룰 때는 값이 어떻게 “보이는가”보다, 실제로 어떤 타입으로 “저장되어 있는가”가 더 중요하다고 느꼈습니다. 

겉으로 보기에는 같은 `1`이어도 숫자형일 수도 있고 문자열일 수도 있으며, 날짜처럼 보이는 값도 실제로는 문자형일 수 있기 때문입니다. 

이 차이를 이해하지 못하면 적절한 함수나 조건식을 사용할 수 없다고 생각했습니다.

강의에서 우선적으로 익혀야 할 대표적인 데이터 타입은 다음과 같았습니다.

| 데이터 타입 | 설명 | 예시 |
| --- | --- | --- |
| 숫자형 | 정수와 실수를 포함하는 수치 데이터입니다. | `1`, `25`, `3.14` |
| 문자열형 | 따옴표로 표현되는 문자 데이터입니다. | `'apple'`, `'2024-01-01'` |
| 날짜/시간형 | 날짜, 시간, 날짜+시간 정보를 담는 데이터입니다. | `DATE`, `DATETIME`, `TIMESTAMP` |
| 불리언형 | 참/거짓을 나타내는 데이터입니다. | `TRUE`, `FALSE` |

특히 중요한 점은, 보이는 값과 저장된 타입이 다를 수 있다는 점입니다. 예를 들어 `2023-12-31`처럼 보여도 실제로는 날짜 타입일 수도 있고 문자열 타입일 수도 있습니다. 

따라서 쿼리를 작성하기 전에 컬럼의 실제 타입을 확인하는 습관이 필요하다고 느꼈습니다.

자료형을 변환할 때는 `CAST`를 사용합니다.

```sql
SELECT CAST(1 AS STRING) AS result;
```

위 쿼리는 숫자 `1`을 문자열 `'1'`로 변환하는 예시입니다. 


즉, `CAST(값 AS 바꿀타입)`의 형태로 사용하는 것입니다. 



하지만 변환이 항상 성공하는 것은 아닙니다. 

예를 들어 `'Kyle'` 같은 문자열을 정수형으로 바꾸려고 하면 오류가 발생합니다.

```sql
SELECT CAST('Kyle' AS INT64) AS result;
```

이 경우에는 숫자로 바꿀 수 없기 때문에 에러가 발생합니다. 이런 상황에서 사용하는 것이 `SAFE_CAST`입니다.

```sql
SELECT SAFE_CAST('Kyle' AS INT64) AS result;
```

`SAFE_CAST`는 변환이 가능하면 값을 반환하고, 불가능하면 오류 대신 `NULL`을 반환합니다. 

실무에서는 데이터가 항상 깨끗하지 않기 때문에, 전체 쿼리가 중단되지 않도록 `SAFE_CAST`를 사용하는 것이 매우 유용하다고 생각했습니다.

정리하면 다음과 같습니다.

| 함수 | 특징 | 변환 실패 시 결과 |
| --- | --- | --- |
| `CAST()` | 일반 형변환 함수입니다. | 오류 발생 |
| `SAFE_CAST()` | 안전하게 형변환하는 함수입니다. | `NULL` 반환 |

또한 강의에서는 `SAFE_DIVIDE`처럼 안전하게 연산하는 함수도 소개되었습니다. 

이를 통해 BigQuery에서는 “실패 가능성이 있는 연산을 안전하게 처리하는 방식”이 중요하다고 느꼈습니다. 

따라서 데이터 타입을 정확히 파악하고, 실패 가능성이 있는 변환에는 `SAFE_CAST`를 적극적으로 활용해야겠다고 생각했습니다.



## 4-3. 문자열 함수(CONCAT, SPLIT, REPLACE, TRIM, UPPER)

~~~
✅ 학습 목표 :
* 문자열 함수들의 종류를 이해하고 어떠한 상황에서 사용하는지 설명할 수 있다. 
~~~

문자열 함수는 텍스트 데이터를 원하는 형태로 가공할 때 매우 자주 사용된다고 느꼈습니다. 

강의에서는 대표적으로 `CONCAT`, `SPLIT`, `REPLACE`, `TRIM`, `UPPER` 다섯 가지 함수를 다루었고, 각각의 목적이 비교적 분명해서 실무에서도 바로 활용할 수 있겠다고 생각했습니다.

또한 Python 과 문법이 비슷하여 이해하기 원활하다고 느꼈습니다. 

먼저 전체적으로 정리하면 다음과 같습니다.

| 함수 | 기능 | 예시 |
| --- | --- | --- |
| `CONCAT` | 여러 문자열을 이어 붙입니다. | 이름 + 성 합치기 |
| `SPLIT` | 특정 기준으로 문자열을 나눕니다. | 쉼표 기준 분리 |
| `REPLACE` | 특정 문자열을 다른 문자열로 바꿉니다. | 오타 수정 |
| `TRIM` | 문자열 앞뒤 또는 특정 문자를 제거합니다. | 공백 제거 |
| `UPPER` | 영문 소문자를 대문자로 바꿉니다. | 대소문자 통일 |

### 1. CONCAT
`CONCAT`은 여러 문자열을 하나로 합치는 함수입니다.

```sql
SELECT CONCAT('안녕', '하세요') AS result;
```

이 함수는 결과를 `안녕하세요`처럼 붙여서 보여줍니다. 여러 컬럼이나 문자열 조각을 이어 붙여서 하나의 문장이나 식별값을 만들 때 유용하다고 생각했습니다.

### 2. SPLIT
`SPLIT`은 특정 문자를 기준으로 문자열을 나누는 함수입니다.

```sql
SELECT SPLIT('가,나,다', ',') AS result;
```

쉼표를 기준으로 문자열을 나누어 배열 형태로 반환합니다. 즉, 한 문자열 안에 들어 있는 여러 값을 분리할 때 사용한다고 이해했습니다. 

다만 쉼표 뒤 공백까지 함께 저장되어 있으면 결과에도 공백이 남을 수 있으므로 구분 기준을 정확히 지정해야 한다고 느꼈습니다.

### 3. REPLACE

`REPLACE`는 문자열 안의 특정 단어를 다른 단어로 바꾸는 함수입니다.

```sql
SELECT REPLACE('안녕하세요', '안녕', '반갑') AS result;
```

결과는 `반갑하세요`처럼 바뀝니다. 잘못된 표현을 수정하거나, 코드값을 다른 표현으로 바꾸고 싶을 때 활용할 수 있다고 생각했습니다.

### 4. TRIM
`TRIM`은 문자열의 앞뒤 공백이나 특정 문자를 제거할 때 사용합니다.

```sql
SELECT TRIM('  hello  ') AS result;
```

이 경우 앞뒤 공백이 제거됩니다. 강의 예시처럼 특정 문자를 잘라내는 방식으로도 사용할 수 있으며, 전처리 과정에서 불필요한 공백을 정리할 때 꼭 필요한 함수라고 느꼈습니다.

### 5. UPPER
`UPPER`는 영문 소문자를 대문자로 바꾸는 함수입니다.

```sql
SELECT UPPER('abc') AS result;
```

결과는 `ABC`입니다. 데이터마다 대소문자가 섞여 있을 때 비교가 어려워질 수 있으므로, 문자열을 같은 기준으로 맞추기 위해 사용하는 함수라고 이해했습니다.

문자열 함수 학습 후 느낀 점은, 문자열 처리는 단순히 보기 좋게 만드는 작업이 아니라 분석 가능한 형태로 정리하는 과정이라는 점입니다. 

예를 들어 공백을 제거하지 않으면 같은 값도 다르게 인식될 수 있고, 대소문자를 통일하지 않으면 조건 비교가 정확히 되지 않을 수 있습니다. 

따라서 문자열 함수는 데이터 전처리의 핵심 도구라고 생각했습니다.



## 4-4. 날짜 및 시간 데이터 이해하기(1) (타임존, UTC, Millisecond, TIMESTAMP/DATETIME)

~~~
✅ 학습 목표 :
* 날짜 및 시간 데이터 타입과 UTC의 개념을 설명할 수 있다. 
* DATE, DATETIME, TIMESTAMP 에 대해서 설명할 수 있다.
* 시간함수들의 종류와 시간의 차이를 추출하는 방법을 설명할 수 있다. 
~~~

날짜와 시간 데이터는 대부분의 서비스 데이터에 반드시 포함된다고 느꼈습니다. 생성 시각, 수정 시각, 주문 시각, 접속 시각처럼 거의 모든 이벤트에는 시간이 붙기 때문입니다. 

그래서 시간 데이터를 정확히 이해하지 못하면 분석 결과 자체가 왜곡될 수 있다고 생각했습니다.

강의에서 가장 먼저 구분한 핵심 타입은 `DATE`, `DATETIME`, `TIMESTAMP`였습니다.

| 타입 | 의미 | 예시 | 특징 |
| --- | --- | --- | --- |
| `DATE` | 날짜만 저장합니다. | `2024-01-01` | 연-월-일 정보만 있습니다. |
| `DATETIME` | 날짜와 시간을 함께 저장합니다. | `2024-01-01 23:59:10` | 타임존 정보는 포함하지 않습니다. |
| `TIMESTAMP` | 특정 시점을 저장합니다. | `2024-01-01 14:00:00 UTC` | UTC 기준 시점이며 타임존과 관련이 있습니다. |

특히 `TIMESTAMP`와 `DATETIME`의 차이를 정확히 이해하는 것이 중요했습니다. 

`TIMESTAMP`는 “UTC 기준의 절대적인 시점”이고, `DATETIME`은 “사람이 읽기 쉬운 날짜+시간 표현”에 가깝다고 이해했습니다. 

따라서 둘은 비슷해 보여도 의미가 다르기 때문에 변환할 때 주의해야 합니다.

### UTC와 타임존
UTC는 국제 표준시라고 이해했습니다. 한국 시간은 일반적으로 `UTC+9`이므로, UTC 기준 시각에 9시간을 더하면 한국 시간이 됩니다. 


실무에서는 이 차이를 놓치면 시간대별 분석이 완전히 잘못될 수 있기 때문에 매우 중요하다고 느꼈습니다.

예를 들어 UTC 기준으로 저장된 시각을 한국 시간으로 보려면 타임존 정보를 함께 반영해야 합니다. 

강의에서는 `Asia/Seoul`을 명시하지 않으면, 의도한 한국 시간이 아니라 UTC 기준 결과로 보일 수 있다고 설명했습니다. 저도 시간 데이터가 이상하게 보일 때는 먼저 타임존 누락을 의심해야겠다고 생각했습니다.

### Millisecond와 Microsecond
강의에서는 초보다 더 세밀한 시간 단위도 다루었습니다.

| 단위 | 의미 |
| --- | --- |
| Millisecond | 1초의 1/1000입니다. |
| Microsecond | 1밀리초의 1/1000입니다. |

실무 데이터에서는 긴 숫자 형태의 시간이 저장되어 있는 경우가 많은데, 이런 값이 밀리초 단위 타임스탬프일 수 있다고 배웠습니다. 

처음 보면 당황할 수 있지만, 이를 적절한 함수로 변환하면 사람이 읽을 수 있는 날짜/시간 형태로 바꿀 수 있다고 이해했습니다.

### 시간 데이터 변환

강의에서는 밀리초를 타임스탬프로 바꾸고, 다시 사람이 보기 쉬운 `DATETIME`으로 변환하는 흐름을 설명했습니다. 


또한 타임존을 반영하지 않으면 9시간 차이가 발생할 수 있으므로, 한국 시간 기준 분석이라면 `Asia/Seoul`을 명시해야 한다고 느꼈습니다.

### 시간 관련 함수에 대한 이해
이번 강의에서 시간 함수의 세부 문법을 아주 많이 다루지는 않았지만, 시간 데이터에서는 크게 두 가지 작업이 중요하다고 이해했습니다.

1. 시간 타입을 서로 변환하는 것  
2. 시간의 일부를 추출하거나 차이를 계산하는 것  

예를 들어 연도, 월, 일, 시를 추출해서 시간대별 분석을 할 수 있고, 두 시점의 차이를 계산해서 경과 시간을 구할 수도 있습니다. 

따라서 시간 데이터는 단순한 기록값이 아니라, 추세 분석과 패턴 분석을 위한 핵심 변수라고 생각했습니다.

마지막으로 제가 이해한 핵심을 정리하면 다음과 같습니다.

| 핵심 포인트 | 설명 |
| --- | --- |
| `DATE` | 날짜만 저장합니다. |
| `DATETIME` | 날짜와 시간을 저장하지만 타임존 정보는 없습니다. |
| `TIMESTAMP` | UTC 기준 특정 시점을 나타냅니다. |
| `UTC+9` | 한국 시간은 UTC보다 9시간 빠릅니다. |
| 타임존 명시 | 한국 시간 분석 시 `Asia/Seoul` 확인이 중요합니다. |
| 밀리초 변환 | 긴 숫자 형태의 시간 데이터를 실제 시각으로 바꿀 수 있습니다. |

이번 범위에서 가장 인상 깊었던 점은, 시간 데이터는 단순히 날짜를 읽는 문제가 아니라 “어느 기준 시각으로 저장되었는가”까지 함께 확인해야 한다는 점이었습니다. 앞으로 시간 관련 분석을 할 때는 값만 보지 않고, 타입과 타임존까지 함께 점검해야겠다고 생각했습니다.

SELECT BOOK_ID, DATE_FORMAT(PUBLISHED_DATE, '%Y-%m-%d') AS PUBLISHED_DATE
FROM BOOK
WHERE CATEGORY = '인문'
  AND YEAR(PUBLISHED_DATE) = 2021
ORDER BY PUBLISHED_DATE ASC;

이 문제에서는 먼저 CATEGORY = '인문' 조건을 사용하여 인문 카테고리에 해당하는 도서만 추출하였습니다. 

그다음 YEAR(PUBLISHED_DATE) = 2021 조건을 사용하여 출판일의 연도가 2021년인 도서만 선택하였습니다. 

마지막으로 ORDER BY PUBLISHED_DATE ASC를 사용하여 출판일 기준 오름차순으로 정렬하였습니다. 또한 문제에서 출판일의 데이트 포맷을 맞춰야 한다고 되어 있으므로 DATE_FORMAT(PUBLISHED_DATE, '%Y-%m-%d')를 사용해 예시와 같은 형식으로 출력하였습니다.

기존 풀이에서 오류가 발생한 이유는 YEAR(PUBLISHED_DATE, 2021)처럼 YEAR 함수에 인자를 2개 넣었기 때문입니다. 

YEAR() 함수는 날짜 데이터 1개만 받아서 해당 날짜의 연도만 추출하는 함수이므로, YEAR(PUBLISHED_DATE)처럼 먼저 연도를 추출한 뒤 = 2021로 비교해야 합니다. 

오류 메시지인 Number of arguments does not match for function YEAR도 바로 이 부분을 알려주는 것이라고 이해했습니다. 

따라서 오류 메시지를 읽고 함수에 들어가는 인자의 개수와 사용법을 먼저 확인하는 것이 중요하다고 느꼈습니다.


## 문제 1

> **🧚Q. 프로그래머스 문제를 풀던 규서는 여러 번의 시행착오 끝에 결국 혼자 해결하기 어려워 오류 메시지를 공유하며 도움을 요청했습니다. 여러분들이 오류 메시지를 확인하고, 해당 SQL 쿼리에서 어떤 부분이 잘못되었는지 오류 메시지를 해석하고 찾아 설명해주세요.**

~~~sql
# 조건에 맞는 도서 리스트 출력하기
# 규서의 SQL 첫 번째 풀이
SELECT BOOK_ID, PUBLISHED_DATE
FROM BOOK
WHERE CATEGORY = '인문'
  AND YEAR(PUBLISHED_DATE, 2021);
  
오류 메시지 : Error: Number of arguments does not match for function YEAR
~~~



~~~
오류 메시지인 `Number of arguments does not match for function YEAR`는 `YEAR` 함수에 전달한 인자의 개수가 맞지 않는다는 뜻입니다. 

즉, `YEAR(PUBLISHED_DATE, 2021)`처럼 두 개의 값을 넣었기 때문에 오류가 발생한 것입니다.

여기서 잘못된 부분은 `YEAR` 함수를 조건식처럼 사용했다는 점입니다. `YEAR` 함수는 날짜값에서 연도만 추출하는 역할을 해야 하는데, 현재 쿼리에서는 `PUBLISHED_DATE`와 `2021`을 동시에 넣어 함수 문법을 잘못 작성했습니다.

의도한 조건은 “출판일의 연도가 2021년인 데이터만 조회하라”는 의미이므로, 먼저 출판일에서 연도를 추출한 뒤 그 결과를 2021과 비교해야 합니다.

BigQuery 기준으로는 다음과 같이 작성하는 것이 적절합니다. 

SELECT BOOK_ID, PUBLISHED_DATE
FROM BOOK
WHERE CATEGORY = '인문'
  AND EXTRACT(YEAR FROM PUBLISHED_DATE) = 2021;

즉, 이 오류는 함수의 의미를 잘못 이해했다기보다는 함수의 입력 방식과 비교식을 한 번에 섞어서 작성했기 때문에 발생한 문법 오류입니다. 

이를 통해 함수는 “값을 추출하는 역할”, 비교 연산자는 “조건을 검사하는 역할”로 나누어 작성해야 한다고 생각했습니다.
~~~
